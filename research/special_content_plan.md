## 二、对每项的决策矩阵

评分维度：
- **F** = 频率（0-3）：0 罕见 / 1 少 / 2 中 / 3 高
- **I** = 对 RAG 影响（0-3）：0 无 / 1 影响 chunk 完整性 / 2 影响检索命中 / 3 影响答案对错
- **C** = 实现成本（0-3）：0 一行代码 / 1 半天 / 2 一天 / 3 多天
- **优先级** = 由 F+I 和 C 综合定

|---|---|---|---|---|---|---|

### 📊 15-query 评测发现的 Known Limitation（2026-07-07）

15 query 覆盖 5 大类（精确数字 / 中英文跨语言 / section 过滤 / 跨公司对比 / 定性描述），13 正确 + 1 部分正确 + 1 失败。两条不修的问题：

#### L1 — Glossy AR 的 section=risk 缺失（C1）

- **现象**：query "MSFT risk factors"（filter section=risk）返回 0 命中
- **第一性原理排查**：MSFT 2022 Annual Report 是 **Glossy AR**（美化版），全文 0 个 "Item N." 标题；"Risk Factors" 只出现在 Forward-Looking Statements 的引用里，实际内容被改写成 MD&A 里的 "Economic Conditions, Challenges, and Risks" 子标题。M4 regex 不命中是**符合原文的正确行为**
- **不修理由**：放宽 regex 会污染标准 10-K 的 section 标签，违反"反对打表修改"原则
- **降级路径**：Glossy AR 用户直接用全文语义检索（不加 section filter），验证过语义 recall 正常
- **影响面**：待统计 101 份里 Glossy AR vs 10-K/10-Q 的占比
- **答辩话术**：section metadata 是**基于 10-K/10-Q 标准结构**的加速通道，对 Glossy AR 类形态降级为全文检索，语义命中不受影响

#### L2 — 数字型 query 优先命中封面/CEO overview（A1）

- **现象**：query "MSFT 2022 total revenue" 命中封面 + CEO letter 的营收提及，未直接命中 Income Statement 数字表
- **根因猜测**：数字表 chunk 里数字密度高、语义描述弱 → embedding 被 "revenue/total" 关键词的散文段落稀释
- **不修理由**：需要调整 chunk 策略（如数字密度加权 / 表格 chunk 单独 embedding），改动量大且可能反噬其他 query
- **降级路径**：LLM 拿到 CEO letter 里的营收数字通常也能答对（Microsoft 2022 revenue $198.3B 在 CEO letter 明确提到）
- **答辩话术**：定性描述 + 数字表两路都覆盖到语义相关内容，LLM 综合作答；如需强制命中数字表，可在应用层加"财务数据类"query 分类器加权 section=financials

**总结**：15 题实际 recall = 14/15 = **93.3%**，1 失败为数据形态问题（Glossy AR），符合预期。

**答辩话术**：
> "我基于 20 份 PDF × 1139 页真实数据做了内容频率调研，把 22 项待定按'频率 × 影响 ÷ 成本'排序，产出 3 层方案：14 项必做、4 项运行时兜底、8 项 known limitation。所有决策都有数据支撑，不是拍脑袋。"
